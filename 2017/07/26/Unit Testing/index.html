前端单元测试

Posted on 2017-07-26 | In Trust The Process

好的测试让开发事半功倍


为什么要做单元测试

单元测试，是指对软件中的最小可测试单元进行检查和验证。

这个最小测试单元，可以是一个 Function ，可以是一个 Object ，也可以是一个 Component ，或者是一个所依赖的 Module。

在目前的工作流中，前端的测试工作主要由测试团队完成，开发团队本身做的测试工作相对较少。目前测试团队的测试方法相对简陋，基本都是简单的人工测试。所以 单元测试 作为开发周期中比较重要的一环，对项目的质量有着非常关键的作用。它属于一种 白盒测试 ，是测试流中最靠前的一个环节，所以这一部分的工作对于我们开发成员来说是很有必要的。

健全的单元测试不仅可以保证核心逻辑的 高可用性 ，还可以作为另一种维度上的 文档 而存在。比如对于项目中很常见的 helpers.js，开发成员只需阅读这个文件的 测试用例 即可明白如何使用。

说到单元测试，顺便提一下 BDD 和 TDD 这两种开发模式。

BDD - 行为驱动开发（Behavior Driven Development），是一种新的敏捷开发方法。它更趋向于需求，需要共同利益者的参与，强调用户故事（User Story）和行为。

TDD - 测试驱动开发（Test Driven Development） 主要遵循以下几个规则：

  • 写一个单元测试去描述程序的一个方面。
  • 运行它应该会失败，因为程序还缺少这个特性。
  • 为这个程序添加一些尽可能简单的代码保证测试通过。
  • 重构这部分代码，直到代码没有重复、代码责任清晰并且结构简单。
  • 持续重复这样做，积累代码。

另外，衡量是否使用了TDD的一个重要标准是测试对代码的覆盖率，覆盖率在80%以下说明一个团队没有充分掌握TDD，当然高覆盖率也不能说一定使用了TDD，这仅仅是一个参考指标。

怎么做单元测试

所以在分清 BDD 和 TDD 之后，针对我们具体的业务需求，前端要做的单元测试工作主要覆盖以下几个方面：

  1. 工具类 .js 的可用性测试，比如 helpers.js 、 filters.js 等。
  2. vuex 中的核心 mutations ，getters 和 actions 的逻辑测试。
  3. 核心组件的逻辑测试。只是逻辑，暂不涉及界面与交互。

针对第一点，测试用例应尽可能的 详尽 。比如 helpers.js 中有一个转换日期格式的 Function ，测试用例更多的是作为 开发文档 的意义而存在，它会告诉开发者，这个函数需要接受什么样的 参数 ，最终会输出什么样的 结果 。所以测试用例越详尽对开发的帮助越大。

针对第二点， vuex 官方给出了详尽的 单元测试文档 。关于 mutations 的测试主要覆盖 store 中一些比较重要的处理逻辑， 关于 actions 的测试主要用来保证后台接口的可用性，数据的准确性。这里面有关 边界情况 的测试很重要。

针对第三点， vue 官方也给出了相应的 单元测试文档 。关于组件的测试相对是比较复杂的，一方面涉及到逻辑的正确性，另一方面还涉及到组件 交互 与 样式 的可用性。考虑到交互与样式的测试工作较为繁重，所以对于组件的测试主要侧重于组件逻辑这一块。后续可以考虑深入。

顺便提一下，第二点和第三点的文档中，都提到了 Karma 这个自动化测试框架，这个也会作为后续的优化方向来进行。

技术选型

测试框架：Mocha

断言库：Chai

Jest 和 Mocha 是目前两个比较流行的轻量级测试框架。两者的生态环境对比如下：

由于 Jest 与 React 的相性更高，所以我们选择 Mocha 。Mocha 自身是不带断言库的，可供选择的断言库有很多，包括 should.js ， expect.js， chai 等。 Chai 是一个同时支持 BDD 和 TDD 风格的断言库，在 node 和浏览器环境都可以运行，相比之下更为高效。

Vue 官方提供的 webpack模板 中，内置了一套 Karma + Mocha + Chai 的自动化测试框架。如果对这三个工具都比较熟悉，可以直接仿照他的方式来进行框架搭建。

其中，一个 describe 描述的就是一个 test suite，这个 suite 主要负责测试日期转换函数的处理逻辑。它里面包含了两个 test case ，每个 test case 用 it 关键字来描述，第一个 case 输入一个合理的日期，然后期望得到一个正确的转换结果，第二个 case 给到一个非法的日期，然后期望得到错误信息的提示。

到这里一个简单的单元测试就算完成了，当然上面的示例还是比较粗糙的，并没有对 特殊值 和 边界值 编写相应的 test case。从这个例子中可以看出，如果关于 日期转换函数 的测试用例写的很详尽的话，开发者可以直接把它当做 文档 来使用，这个函数怎么用，看测试用例就可以一目了然了。

后续优化方向

  1. 如果后续测试工作开展顺利，且越来越重要的情况下，可以考虑加入 karma 的 自动化测试框架 。karma 可以模拟不通浏览器的测试环境，可以大大提升 浏览器兼容性 和 设备兼容性 的测试效率。
  2. 目前的测试并不包含界面测试，后续可以考虑加入一些 核心样式 的相关测试。
