使用 Webpack 进行项目构建

Posted on 2017-07-20 | In Trust The Process

Webpack 项目构建是性能优化的重要环节


Webpack简介

微社区项目使用 Webpack 来进行项目构建。

Webpack能够很好地管理、打包Web开发中所用到的HTML、Javascript、CSS以及各种静态文件（图片、字体等）。对于不同类型的资源，Webpack有对应的模块加载器，比如Vue组件通过 vue-loader 来进行编译和加载。

Webpack通过分析模块间的依赖关系，最后生成优化且合并后的静态资源。

Webpack的优势

  • 两大特色：Code Splitting 以及可以支持串联操作的 Loaders 。
  • 具有 RequireJS 和 Browerify 的功能，比前两者更加强大。
  • 对 CommonJS、AMD、ES6的语法做了兼容。
  • 具有强大的 Plugin 系统，大多是内部插件，使用起来比较灵活。

Webpack相关配置

Webpack通过配置文件的形式对项目构建进行规则的定义，如果你是第一次接触Webpack，可以先熟悉一下 Webpack的核心概念 。如果你已经对Webpack很熟悉，可以参考 RapidVueES6 项目的相关配置文件，了解一下目前项目中所使用到的Webpack功能，其中包括了 代码分离 ， 代码编译 ， 代码压缩， ES6+语法转换 等等。下面会以RapidVueES6项目为例，对一些核心功能进行说明。

代码分离

代码分离是 webpack 中最引人注目的特性之一。你可以把你的代码分离到不同的 bundle 中，然后你就可以去按需加载这些文件。总的来说，使用 webpack 可以完成两类代码分离工作：资源分离（Resource Splitting）以及代码按需分离（On Demand Code Splitting）

详细的代码分离介绍，可以参考 官方给出的相关定义 。这里主要介绍一下RapidVueES6的具体实践方式。

  • 定义entry

    这两个entry中， vendor 包含了项目中用到的所有第三方库， main 指向了项目的业务代码入口。定义了这两个 entry 之后，webpack会把所有的第三方库打包成一个 vendor.js ，所有的业务代码打包成一个 main.js 。这样把代码进行分离的 目的 在于，业务代码的 变更频率 往往比第三方库高的多，而对于浏览器来说，变更频率相对很低的 vendor.js 是可以进行长期缓存的，一定程度上提高了页面的 加载性能 。

  • 定义output

    这个字段定义了所有的入口文件最终的输出形式。

    • path：定义了打包后的输出目录。
    • publicPath：定义了对于浏览器而言项目的根目录位置。代码中的 相对URL 都以这个publicPath作为参照系。
    • chunkFilename：定义了非入口 chunk 文件的名称。除了入口文件以外的模块，都称之为 chunk 。
    • filename：定义了入口文件的名称。

    其中需要注意的是，在两种不同的项目构建模式下，chunkFilename 和 filename 是不同的。在生产模式中，文件名加入了md5 hash值，如果文件内容没变，文件名就不会发生变化，有助于浏览器的缓存。

  • 定义plugins

    webpack提供强大的plugins系统。有一些是webpack内置的，也有一些是丰富的webpack生态圈所提供的。

  • 代码按需分离

    编写代码的过程中，常常碰到按需加载的相关需求。在实际的使用场景中，页面往往只需要加载用户所访问的路由的相关资源，这很大程度上提高了页面的首次加载性能。

    在实际的实现方案中，只需在定义vue-router的路由信息时，用require.ensure的方式加载相应组件即可。

  • 打包截图

从打包的结果中可以看到，除了 main 和 vendor 以外，还有 index 、 usercenter 、 strategy 另外三个独立的chunk。需要注意的是，这里面有一个 manifest 模块，是在 CommonsChunkPlugin 中所定义的，里面记录了所有模块的索引信息。可以提升webpack的打包速度。

代码编译

module 字段定义了一系列与代码编译相关的工具和规则。里面每个属性的含义可以在 官方给出的文档 中查到。以 RapidVueES 为例，展示了几个比较常用的属性及其相关注释。

具体说明一下上述不同类型资源的解析方式：

  • .vue : 采用 vue-loader 将 vue 组件转换为 JavaScript 模块。它会解析文件，提取每个语言块，如有必要会通过其它 loader 处理，最后将他们组装成一个 CommonJS 模块。
  • .js : 采用 babel-loader 转换 ES6+ 的语法，配合 .babelrc 一起使用。
  • .png|.jpg|.gif|.svg : 采用 file-loader 输出图片资源的 public path。详情参考 File-loader说明 。
  • .css : Webpack允许像加载任何代码一样加载 CSS。加载 CSS 需要 css-loader 和 style-loader，他们做两件不同的事情，css-loader会遍历 CSS 文件，然后找到 url() 表达式然后处理他们，style-loader 会把原来的 CSS 代码插入页面中的一个 style 标签中。

模块解析

Webpack 通过 resolve 字段来设置模块如何被解析。常见的配置项有：

  • alias : 创建 import 或 require 的别名，来确保模块引入变得更简单。
  • modules : 告诉 webpack 解析模块时应该搜索的目录。
  • extensions : 自动解析确定的扩展。默认值为：[".js", ".json"]。能够使用户在引入模块时不带扩展。

杂项

  • devServer 字段定义了与 webpack-dev-server 相关的一些选项。包括了 模块热替换 以及 publicPath 的指定等。
  • devtool 字段用来控制是否生成，以及如何生成 source map 。
    • 线上环境 应该设置 devtool: false ，否则会暴露源码。如果需要在真实环境中进行 断点调试 ，应该在 测试环境 中进行，而不是正式的 线上环境 。
    • eval-source-map : 适用于 开发环境 ，每个模块使用 eval() 执行，并且 SourceMap 转换为 DataUrl 后添加到 eval() 中。初始化 SourceMap 时比较慢，但是会在重构建时提供很快的速度，并且生成实际的文件。行数能够正确映射，因为会映射到原始代码中。
    • source-map : 适用于 生产环境 ，生成完整的 SourceMap，输出为独立文件。由于在 bundle 中添加了引用注释，所以开发工具知道在哪里去找到 SourceMap。

结语

目前webpack团队已经发布了最新的3.0版本，总体来说3.0与2.0是完美兼容的，没有那种革命性的变化。官方给出的说法是：

Migrating from webpack 2 to 3, should involve no effort beyond running the upgrade commands in your terminal.

所以像类似于webpack这类效率型的构建类工具，个人认为掌握其用法和使用技巧是最重要的。
